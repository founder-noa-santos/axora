R-06 Findings: Agent ArchitectureResearch Date: 2026-03-16Researcher: Expert AnalystSources: The paradigm of artificial intelligence in software engineering has irrevocably shifted from isolated, single-turn completion models to orchestrated, multi-agent systems (MAS). Single-agent approaches consistently fail when confronted with enterprise-scale codebases due to context window exhaustion, inability to self-correct complex logic, and a lack of role-specific focus. Multi-agent systems address these deficiencies by fracturing complex software development life cycles (SDLC) into discrete operations managed by specialized Large Language Model (LLM) instances. This report exhaustively analyzes the orchestration architectures, frameworks, task assignment algorithms, state management protocols, and conflict resolution strategies necessary to construct AXORA, an optimal multi-agent AI coding system.Deep Research: Orchestration ArchitecturesThe architectural foundation of a multi-agent system dictates how information propagates, how computational resources are allocated, and how divergent outputs are synthesized into a coherent final product. In the context of software engineering, where tasks exhibit high degrees of interdependency and strict syntactic constraints, the choice of orchestration pattern is the primary determinant of system reliability and scalability.Centralized vs. Decentralized vs. HierarchicalCentralized orchestration, predominantly implemented as the Orchestrator Pattern, relies on a single distinguished agent or control plane to route messages, manage state, and dictate the execution flow. Frameworks employing this topology offer exceptionally high observability and simplified debugging, as all inter-agent communication must pass through the central node. The orchestrator maintains the global state and decides which specialized agent should execute next based on the current context. However, this architecture introduces a severe computational and cognitive bottleneck. The central orchestrator must process every message, leading to rapid context window saturation and escalating token costs. If the orchestrator fails to correctly parse the current state of the software repository, the entire system is paralyzed.Decentralized, or peer-to-peer (P2P), architectures distribute the coordination logic entirely across the agents themselves. In this model, agents communicate directly using graph-based routing or broadcast message-passing protocols without a central authority. While this maximizes resilience and eliminates the single point of failure, it significantly increases the complexity of the system. Unregulated peer-to-peer communication often results in infinite loops, redundant code generation, and "conversation drift," where agents lose track of the original software requirements. Decentralized networks require strict communication protocols, such as the Agent-to-Agent (A2A) protocol, which utilizes cryptographically signed, structured metadata to govern negotiation and delegation. For software engineering, pure P2P networks are generally too chaotic to produce deterministic, compilable code reliably.Hierarchical orchestration strikes a pragmatic balance by organizing agents into tree-like structures resembling traditional corporate topologies. In this model, manager agents oversee specific sub-domains. For example, a Frontend Manager oversees UI and State agents, while a Backend Manager oversees Database and API agents. The manager allocates tasks based on the localized capabilities of its subordinates and validates their outputs before passing the synthesized result up the chain of command. This pattern limits context pollution, as higher-level managers only ingest the summarized outputs of lower-level teams rather than granular execution logs, making it highly suitable for complex, multi-repository software projects.Blackboard ArchitectureThe Blackboard architecture, a revitalization of classical artificial intelligence paradigms, has emerged as a highly effective solution for LLM orchestration. In this pattern, agents do not communicate directly with one another via message passing. Instead, they read from and write to a shared memory space known as the blackboard. A continuous evaluation process monitors the blackboard to determine when a consensus has been reached or a subtask has been completed.Modern implementations, such as the Large Language Model Multi-Agent System (LbMAS), demonstrate that the blackboard architecture drastically reduces token overhead compared to centralized message-passing. The system consists of a control unit, the shared blackboard (often divided into public and private namespaces), and a pool of heterogeneous agents. The control unit dynamically selects which agents to activate during each round based on the current state of the blackboard. Because agents self-select or are selected based on the evolving problem state rather than a rigid predefined workflow, the system handles ill-structured problems—like discovering bugs in undocumented legacy code—with remarkable efficiency. By decoupling the agents from direct interdependency, the blackboard architecture elegantly solves the context fragmentation problem that frequently plagues distributed coding systems.Pipeline ArchitecturePipeline architectures arrange agents in a strict linear sequence, where the structured output of one agent becomes the immutable input for the next. This pattern is exceptionally efficient for deterministic, well-defined workflows, such as a CI/CD automation sequence (e.g., Code Generation $\rightarrow$ Static Analysis $\rightarrow$ Unit Testing $\rightarrow$ Deployment). However, pipelines are inherently rigid. They lack the capacity for bidirectional feedback loops. If the Unit Testing agent discovers a syntax error, a pure pipeline architecture cannot seamlessly route the context back to the Code Generation agent for refinement without breaking the architectural contract. Therefore, pure pipelines are generally unsuitable for the highly iterative, collaborative nature of exploratory software development, though they remain valuable for final code-verification phases.Market-Based ArchitectureMarket-based architectures allocate tasks through computational economic mechanisms, specifically auction protocols. In systems like the Agent Exchange (AEX), a central auction engine translates human objectives into executable tasks, and available agents submit bids indicating their capability, availability, and estimated token cost to complete the task.This decentralized allocation method organically optimizes resource distribution, ensuring that a highly capable but computationally expensive model is not wasted on trivial formatting tasks, which can be won by smaller, faster models via lower bids. A cost-effectiveness metric evaluates both the bid and the agent's historical fitness for the task. While mathematically elegant and highly scalable, market-based systems introduce substantial orchestration overhead. The system must continuously calculate revenue expectations, process multi-round bidding, and maintain capability representations for every agent, making it overly complex for small-scale, localized code generation but highly attractive for enterprise-wide, cloud-deployed agent ecosystems.Multi-Agent Frameworks AnalysisThe selection of an underlying framework dictates the constraints and capabilities of the multi-agent environment. Lessons for AXORA:What to adopt: The clear separation of concerns between reasoning (the AssistantAgent) and execution (the UserProxyAgent sandboxed in Docker). This prevents the LLM from inadvertently executing malicious code on the host machine.What to improve: AutoGen's reliance on appending the entire conversation history to the prompt rapidly saturates context windows. AXORA must implement a superior state management system (like a Blackboard) that stores only the current Abstract Syntax Tree (AST) and the latest error logs, rather than the entire chat transcript. Additionally, string-matching termination conditions ("TERMINATE") are brittle; structured JSON outputs should govern state transitions instead.CrewAI Deep-DiveCrewAI abstracts agent interaction through an organizational, role-based lens rather than pure dialogue. CrewAI supports two primary orchestration models: sequential and hierarchical. In the sequential process, tasks are processed linearly, with the output of one task explicitly serving as the context for the next. In the hierarchical process, a manager agent—driven by a designated manager_llm—dynamically allocates tasks to subordinate agents based on their predefined capabilities, oversees their execution, and validates the final output before the process can continue.Memory Sharing & Task Assignment:
Unlike AutoGen, which relies heavily on conversation history, CrewAI integrates role-based memory with Retrieval-Augmented Generation (RAG) support. Tasks are treated as distinct objects (Task class) with defined inputs, expected outputs, and assigned tools. In the hierarchical model, task assignment is dynamic; the manager evaluates the capabilities of the crew and routes the task to the most appropriate agent at runtime.Comparison to AutoGen:While AutoGen excels at open-ended, highly iterative dialogue (ideal for pair programming), CrewAI excels at structured, multi-disciplinary workflows (ideal for passing a PRD to an Architect, then to a Coder, then to a Documenter). AutoGen is conversational; CrewAI is procedural.LangGraph (LangChain) Deep-DiveLangGraph approaches multi-agent orchestration fundamentally as a state machine. It models workflows as stateful, cyclic graphs where nodes represent functions (or LLM invocations) and edges represent transitions, often controlled by conditional routing functions.State Machine and Graph-Based Orchestration:
LangGraph operates on discrete iterations called "super-steps," inspired by the Pregel system, where all parallel nodes in a step must complete before the global state updates. Execution is deterministic. The state object is passed between nodes, mutated, and returned.Cycles, Branches, and Persistence:
Cycles are managed naturally via conditional edges, allowing an agent to loop back to a previous node (e.g., returning to a "Code Generation" node if a "Test Validation" node fails). LangGraph's most profound advantage is its built-in persistence layer. By attaching a checkpointer (such as a PostgreSQL backend), LangGraph achieves durable execution. The exact state of the graph is saved at every super-step boundary. This enables "time-travel" debugging, allowing developers to rewind the agent to a previous state, modify the context manually, and resume execution.Comparison to Message-Passing:Traditional message-passing (like AutoGen) relies on emergent behavior driven by the LLM's natural language responses. LangGraph's state machine approach enforces rigorous, programmatic boundaries. The LLM dictates the content of the state, but the framework strictly dictates the flow of execution, resulting in significantly higher reliability for enterprise applications.OpenAI Agents SDK (Swarm)Evolving from the experimental Swarm library, the OpenAI Agents SDK focuses on stateless, lightweight coordination using "handoffs" as the primary orchestration primitive.Handoff Patterns & Context Transfer:
Instead of a central orchestrator or a complex graph, an agent in this SDK is simply an LLM equipped with instructions and tools. One of those tools is the ability to transfer control to another agent. When an agent executes a handoff, it returns the target agent object and optionally updates the context variables. The framework maintains no persistent state between calls, mandating that every handoff includes all the necessary context for the next agent.What we can learn:Explicit handoffs prevent the infinite loops often seen in implicit coordination systems. However, the burden of context management rests entirely on the developer. For AXORA, the handoff pattern is useful for transferring control between sub-routines (e.g., Coder handing off to Documenter), but the statelessness is highly detrimental for managing large codebases.CamelAICamelAI utilizes a strict role-playing architecture to govern agent interaction. It relies on a technique called "inception prompting" to enforce role adherence, systematically eliminating the phenomenon of "role-flipping" where agents forget their assigned persona.Communication Protocols:
The architecture pairs an AI User (acting as the domain expert or task specifier) with an AI Assistant (the executor or coder). The AI User continuously provides targeted instructions, and the AI Assistant provides the corresponding code implementations in a controlled loop. For software engineering, CamelAI's rigorous prompting protocols demonstrate superior performance on benchmarks by preventing agents from engaging in unproductive, open-ended dialogue, forcing them instead to adhere strictly to the instruction-solution format.Task Assignment StrategiesThe algorithmic approach used to distribute programming tasks among available agents heavily influences the system's efficiency and the quality of the generated code.Static AssignmentStatic assignment relies on hardcoded, deterministic roles. An agent designated as the "Database Engineer" will consistently receive all tasks tagged with database modifications. While deterministic, highly predictable, and simple to debug, this strategy lacks the elasticity required to handle varying workloads or ambiguous tasks that cross domain boundaries.Dynamic AssignmentDynamic assignment evaluates the current operational context before distributing work. This involves tracking agent availability, current token load, and context-window saturation. If the primary frontend agent is currently processing a massive UI refactor, the orchestrator dynamically assigns a new component task to an idle, secondary frontend agent. This maximizes parallel execution throughput but requires sophisticated queuing mechanisms.Capability-Based & Difficulty-Aware AssignmentCapability-based routing matches the semantic requirements of a task against a centralized registry of agent capabilities. A cutting-edge evolution of this concept is Difficulty-Aware Agentic Orchestration (DAAO). In this paradigm, the system dynamically adapts the workflow depth and the specific LLM assigned based on the mathematical difficulty of the software query.DAAO utilizes a variational autoencoder to map the input query into a latent representation, outputting a scalar difficulty value. Based on this signal, a modular allocator constructs a specific Directed Acyclic Graph (DAG) for the task. Simple tasks (e.g., writing a boilerplate getter/setter) are assigned to lightweight, fast models executing a single-step workflow. Complex tasks (e.g., implementing a distributed consensus algorithm) trigger the generation of a deep, multi-stage workflow utilizing high-capacity models configured for iterative refinement. Implementing this in AXORA involves extracting metadata from the Jira/GitHub issue, running a lightweight classification model to score the complexity, and routing the task to either a fast, single-shot execution node or a deep, multi-agent debate node.Learning-Based AssignmentLearning-based assignment tracks historical performance metrics to optimize future routing. By analyzing which agents (or underlying models) historically produce code with the highest test-pass rates or the lowest cyclomatic complexity for specific domains, the orchestrator updates a probabilistic routing table. Over time, the system learns that Model A excels at writing test cases, while Model B is superior at Rust memory management.Conflict ResolutionWhen multiple agents manipulate the same codebase, contradictions are inevitable. The CooperBench evaluation—a rigorous study of collaborative coding agents across 600+ tasks—identified that adding agents to a task can actually degrade performance by up to 30% due to "the curse of coordination". Agents frequently suffer from divergent architectural choices, leading to "merge hell".Resolution Strategies for Coding AgentsConflict Scenario 1: Divergent ImplementationsTwo agents working in parallel suggest fundamentally different implementations for overlapping files.Recommended Approach (Merge Strategy): Naive git merge or union strategies (which accept both sides of a conflict) destroy syntax validity. Instead, an explicit Conflict-Resolver agent must be utilized. This agent analyzes the Abstract Syntax Tree (AST) of both proposed modifications and synthesizes a unified implementation that preserves the semantic intent of both agents while maintaining syntactic validity.Conflict Scenario 2: Reviewer Rejects Writer's CodeA Reviewer agent repeatedly rejects the Coder's implementation, creating an infinite revision loop.Recommended Approach (Arbitration): When a deadlock threshold is reached (e.g., 3 failed revisions), a designated Arbiter agent (equipped with a superior LLM, such as Claude 3.5 Sonnet or GPT-4o) is invoked. The Arbiter ingests the AST, the Coder's logic, and the Reviewer's critiques, rendering a final, immutable decision that forces the workflow forward.Conflict Scenario 3: Disagreement on ArchitectureAgents disagree on high-level design patterns (e.g., using an Actor model vs. a standard asynchronous queue).Recommended Approach (Weighted Voting): Solicit opinions from multiple agents, applying a weighted score based on expertise. The Architect agent carries an 80% weight on structural design, overriding the standard Coder's preference.Human-in-the-Loop (HITL) Escalation PathFully autonomous execution is perilous. Effective systems implement threshold-based HITL fallback escalations. If the Arbiter agent's confidence score in resolving an AST conflict falls below a strict threshold (e.g., 85%), or if a compilation error persists after 5 autonomous retry loops, the execution graph suspends. The system serializes the exact state to a database and alerts a human developer. pub struct AstPatch {
    pub author_id: String,
    pub target_file: String,
    pub proposed_code: String,
    pub confidence_score: f32,
}

pub enum ConflictResolution {
    Merged(String),
    EscalateToHuman(String),
}

pub struct ArbitrationEngine {
    pub arbiter_llm_client: DefaultLlmClient,
    pub max_retries: u8,
}

impl ArbitrationEngine {
    pub async fn resolve_merge_conflict(
        &self, 
        patch_a: AstPatch, 
        patch_b: AstPatch
    ) -> ConflictResolution {
        // Step 1: Check confidence thresholds
        if patch_a.confidence_score < 0.85 && patch_b.confidence_score < 0.85 {
            return ConflictResolution::EscalateToHuman(
                "Both agents exhibit low confidence. Manual merge required.".to_string()
            );
        }

        // Step 2: Construct arbitration prompt with AST context
        let prompt = format!(
            "You are a Senior Rust Staff Engineer. Two agents have submitted conflicting patches for {}. \
             Patch A: {} \
             Patch B: {} \
             Synthesize a single, compilation-ready Rust implementation that merges the intent of both patches without violating borrow checker rules.",
            patch_a.target_file, patch_a.proposed_code, patch_b.proposed_code
        );

        // Step 3: Invoke LLM to synthesize the merge
        let synthesis_result = self.arbiter_llm_client.generate(&prompt).await;
        
        match synthesis_result {
            Ok(merged_code) => ConflictResolution::Merged(merged_code),
            Err(e) => ConflictResolution::EscalateToHuman(format!("Arbiter failed: {}", e)),
        }
    }
}
Collaboration PatternsPair Programming Pattern (Driver + Navigator)Adapted from human agile methodologies, this pattern involves two agents working synchronously. The "Navigator" agent focuses on high-level algorithmic planning, boundary conditions, and multi-plan exploration. The "Driver" agent focuses purely on translating the Navigator's plan into syntactically correct code. Implemented via LLMs, the Navigator outputs an abstract plan and specific constraints into a shared memory buffer, which the Driver reads to constrain its code generation trajectory. The Navigator continuously evaluates the Driver's output against the overarching constraints, preventing localized optimization traps.Review PatternThe Review pattern explicitly decouples generation from verification. A Coder agent generates the implementation, which is subsequently passed to an independent Reviewer agent. The AgentCoder framework demonstrates the efficacy of this by separating source code generation from test case generation. A Test Designer agent independently writes edge-case tests without ever seeing the Coder's implementation, ensuring total objectivity. A Test Executor agent runs the code against the tests in a terminal sandbox and feeds the exact error logs back to the Coder for iterative refinement, achieving massive increases in benchmark pass rates.Divide and ConquerFor repository-scale tasks, the problem must be fractalized. A Project Manager agent dissects a feature request into discrete, isolated subtasks. These subtasks are distributed to specialized agents executing in parallel. A crucial requirement for this pattern is Git Worktree isolation, allowing each agent to manipulate and test a separate branch of the codebase simultaneously without creating file-locking conflicts. Finally, an Integration agent merges the parallel branches.Relay PatternAgents operate sequentially, where each agent adds a specific layer of refinement to the codebase. An Architect stubs out the traits and interfaces, passing the files to a Coder who implements the logic, who then passes it to a Documenter who adds inline documentation. This maximizes the specific instruction-following capabilities of each specialized agent.Collaborative DebuggingWhen a complex bug is detected, a triad of agents collaborates: a Diagnoser agent analyzes stack traces and application logs to identify the root cause; a Coder agent generates the patch; and a Verifier agent runs a newly generated regression test to ensure the bug is eradicated and no new regressions are introduced.State ManagementManaging the context and memory of multiple autonomous actors is the most complex engineering challenge in multi-agent orchestration. Memory fragmentation leads to hallucinations, while unbounded context accumulation leads to exponential token costs and context window collapse.Shared State vs. Private StateSystems must rigorously delineate between global and local memory.Private State: Each agent maintains a localized scratchpad for its internal iterative inference, Chain-of-Thought (CoT) reasoning, and transient tool outputs. This prevents the global context from being polluted with intermediate, failed code snippets.Shared State: The global repository containing verified facts, overarching system architecture constraints, and finalized code implementations. In a Blackboard architecture, this is the central data structure.State Synchronization and Concurrency ControlWhen multiple agents attempt to read or modify the shared state simultaneously, the system risks race conditions and state corruption. The SagaLLM framework introduces a transactional approach to LLM state management, dividing memory into Application State ($S_A$), Operation State ($S_O$), and Dependency State ($S_D$).Instead of traditional, heavyweight database locking—which paralyzes asynchronous agent execution—multi-agent coding systems should utilize optimistic concurrency control and compensable transactions. If Agent A and Agent B introduce conflicting updates to the codebase, the Dependency Tracker identifies the logical violation. The system triggers a "compensating transaction," effectively rolling back Agent B's operation and prompting it to regenerate its solution using Agent A's newly committed code as an immutable constraint. State synchronization should be event-driven; agents subscribe to modifications on specific files and pull updates only when a verified merge event occurs.Scaling ConsiderationsHow Many Agents?Scaling multi-agent systems is not a linear equation; more agents do not unilaterally equal better performance. Empirical analyses indicate a sharp diminishing return curve. Adding agents increases the sheer volume of message passing, leading to a "coordination overhead" that rapidly outweighs the benefits of parallelization. Furthermore, excessive agent counts can induce psychological reactance or "groupthink" phenomena within the LLM instances, where agents blindly agree with a vocal majority rather than rigorously verifying the code. For software engineering workflows, the optimal topology generally consists of 3 to 5 highly specialized agents per specific subtask domain (e.g., 1 Coder, 1 Test Designer, 1 Executor, 1 Reviewer).Agent SpecializationAgents must be explicitly bounded in their capabilities. A "Generalist" agent is prone to hallucination when navigating massive repositories. Specialization ensures that the agent's system prompt is highly concentrated on a single domain, maximizing the LLM's attention mechanism. Roles such as Code Writer, Code Reviewer, Test Writer, Documentation Writer, Debugger, and Architect ensure that cognitive load is distributed effectively.Resource ManagementResource management must be strictly enforced at the orchestration layer. Token budgets and hard timeouts must be established for every node.
