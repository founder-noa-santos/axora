Context Compacting, Sharing, and Distribution for Multi-Agent Systems1. The Context Escalation Crisis in Multi-Agent SystemsThe evolution of Large Language Models (LLMs) has catalyzed the transition from single-turn conversational interfaces to long-horizon, autonomous multi-agent systems capable of executing complex, multi-step workflows. However, this architectural leap exposes a critical infrastructural bottleneck: the phenomenon of context escalation. As agents operate over dozens or hundreds of sequential turns, they accumulate massive volumes of conversational history, tool execution logs, retrieved document segments, and intermediate reasoning traces.Without active intervention, the context window saturates rapidly—typically within 10 to 20 operational turns. This saturation triggers a cascade of systemic failures. Economically, token costs explode; processing a 150,000-token context across a swarm of agents can incur expenses ranging from $0.03 to $0.12 per 1,000 tokens per inference step, rapidly rendering production deployments financially unviable. Computationally, latency increases linearly or quadratically due to the fundamental constraints of transformer attention mechanisms, which must compute $O(n^2)$ pairwise relationships for $n$ tokens.Most critically, cognitive degradation occurs. Empirical studies demonstrate that as context volume scales, models suffer from "context rot" or the "lost in the middle" phenomenon. For example, evaluations indicate that model accuracy can plummet from 98.1% to under 64% simply due to the accumulation of irrelevant historical noise. To enable viable, long-running multi-agent systems—such as the envisioned AXORA platform—architecture must transition from passive token accumulation to active context engineering, leveraging advanced compacting, targeted distribution, and mathematically rigorous consistency models.2. Part 1: Context Compacting2.1 Defining Context CompactingContext compacting is the algorithmic and structural reduction of an LLM's operational memory to its maximum information density, discarding syntactic noise and obsolete transactional history while strictly preserving the semantic signals required for future reasoning.It is vital to distinguish compacting from traditional summarization. Summarization is inherently an abstractive, lossy process designed for human readability; it routinely strips away low-level technical specifics (e.g., file paths, exact API parameters, error codes) that an autonomous agent requires to continue a workflow. Compacting, conversely, is an operational preservation strategy. It can be implemented as either a lossless structural transformation (e.g., schema minification) or a highly calibrated lossy compression (where the "loss" only applies to information mathematically proven to have zero utility for the current objective).The ultimate goal of context compacting is to achieve compression ratios of 10:1 to 30:1, effectively decoupling the agent's functional memory from its raw token footprint, thereby preserving reasoning quality while radically accelerating inference.2.2 Core Compacting TechniquesCurrent state-of-the-art approaches to context compacting operate across four distinct paradigms: Summarization-based, Memory-based, Structural, and Attention-based techniques.A. Summarization-Based CompactingWhile naive summarization is flawed, structured, agentic summarization remains highly effective when constrained.Rolling Summary: The system maintains a sliding window of the most recent $N$ turns verbatim. When the threshold is exceeded, the oldest turns are recursively summarized and appended to a running "historical context" block. This prevents sudden context amnesia but can suffer from semantic drift over hundreds of iterations.Hierarchical Summary (H-MEM / HiMem): Advanced systems utilize hierarchical structures that separate episodic memory from semantic knowledge. Interaction logs are processed through a "Topic-Aware Event-Surprise Dual-Channel Segmentation" pipeline, which identifies key events and extracts them into higher-level knowledge nodes, discarding the conversational wrapper.Key-Point (Extractive) Pruning: Rather than rewriting text (abstractive), the system explicitly identifies and retains only the critical factual sentences, preserving the exact syntax of the original data, which is highly beneficial for code and technical parameters.B. Memory-Based CompactingInspired by computer operating systems, memory-based compacting treats the LLM context window as constrained Random Access Memory (RAM) and utilizes external databases as disk storage.Working Memory: A fixed-size, strictly managed segment of the prompt where the agent actively reads and writes immediate task variables.Episodic Memory: Raw, chronological interaction logs that are immediately flushed from the context window and stored externally. They are retrieved solely on an on-demand basis.Semantic Memory: Facts, rules, and world-states extracted from interactions, embedded, and stored in vector databases (e.g., ChromaDB), injected into the prompt only when semantically relevant to the current operational step.C. Structural CompactingStructural compacting eliminates token waste inherent in standard data serialization formats.Schema-Based Pruning: Trimming large JSON payloads by removing null fields, deeply nested irrelevant arrays, or verbose keys.TOON (Token-Oriented Object Notation): JSON is notoriously token-heavy due to repeated keys, brackets, and quotes. TOON strips structural syntax by defining a schema once and streaming only the comma-separated values, reducing the token footprint of repetitive data structures by up to 80% without losing any functional data.Diff-Based Context: Instead of rewriting a full state object into the context, the system provides only the diffs (e.g., RFC 6902 JSON Patches) of what has changed since the last turn, drastically compressing the delta.D. Attention-Based and Latent CompressionThese methodologies manipulate how the transformer model physically processes tokens.Hard Compression (Token Pruning): Frameworks like LLMLingua utilize smaller, specialized language models to calculate the information entropy or perplexity of every token in a prompt. Tokens with low entropy (highly predictable, redundant words) are physically removed. While highly efficient, aggressive hard compression risks destroying long-range semantic dependencies.Soft Compression (Latent Context Compilation): The current frontier of compacting. Frameworks compile vast contexts directly into latent embeddings (KV cache tensors) rather than text tokens. Utilizing techniques like In-context Autoencoders (ICAE) or disposable Low-Rank Adaptations (LoRAs), massive documents are distilled into a small set of continuous "Buffer Tokens." Experiments demonstrate this achieves 16x to 32x compression ratios with near-zero loss in fine-grained reasoning fidelity, entirely bypassing the text-tokenization bottleneck.2.3 The Size-Fidelity Paradox and ACONRecent research highlights a critical phenomenon in context compacting: the "Size-Fidelity Paradox." When deploying large foundation models to compress context, the models frequently hallucinate, overwriting the source facts with their own pre-trained priors (semantic drift). Consequently, the industry is shifting toward smaller, distilled models specifically optimized for compression.A leading framework in this space is Agent Context Optimization (Acon), which provides a unified pipeline for compressing both environment observations and long-horizon interaction histories. By utilizing failure-driven, task-aware guidelines, Acon distills histories safely, reducing peak token usage by 26% to 54% while actually improving the downstream reasoning accuracy of the multi-agent system by up to 46%.Table 1: Context Compaction Techniques EvaluationTechnique CategorySpecific MechanismAchievable CompressionReasoning PreservationImplementation ComplexitySummarizationHierarchical / Rolling (H-MEM)5x - 10xModerate (Risk of semantic drift)Low to ModerateMemory-BasedOS-Style Paging (Letta)Infinite (Fixed Window)High (Requires robust retrieval)HighStructuralTOON / JSON Patch2x - 5xPerfect (Lossless)LowAttention (Hard)Entropy Pruning (LLMLingua)3x - 10xModerate (Syntax degradation)High (Requires secondary model)Attention (Soft)Latent Compilation (KV Cache)16x - 32xVery High (Preserves latent facts)Very High (Requires architecture modification)3. Part 2: Context Sharing & DistributionA multi-agent system relies on the seamless exchange of insights, states, and plans between autonomous nodes. However, providing every agent with the full global context causes immediate exponential token explosion. Sharing architectures must be optimized to distribute only the precise context required, exactly when it is required.3.1 Multi-Agent Context ArchitecturesIndustry standard orchestration relies on three distinct architectural topologies to govern inter-agent knowledge exchange.A. Centralized Blackboard ArchitectureThe blackboard pattern operates as a shared, highly observable knowledge base. A central "war room" state is maintained, where agents read current global objectives and asynchronously write their specialized findings back to the board.Maintenance: The blackboard is hosted on an in-memory data store (e.g., Redis) or a real-time database.Subscription: Agents monitor the board for state changes matching their specific preconditions.Strengths: Exceptional for opportunistic, non-linear problem solving and semantic reasoning. It completely decouples agents, as they do not need to know of each other's existence.Weaknesses: Highly susceptible to race conditions and read/write conflicts if concurrency is not strictly managed.B. Publish-Subscribe (Pub-Sub) ArchitectureIn the pub-sub topology, the system functions as a reactive, event-driven mesh. There is no central intelligence orchestrating the data flow.Mechanism: When an agent updates its local state or completes a task, it publishes a standardized event payload to a message broker (e.g., NATS, Redis Pub/Sub, Kafka).Filtering: Other agents subscribe exclusively to topics relevant to their roles (e.g., a Quality Assurance agent subscribes only to code_commit topics).Strengths: Highly scalable, minimizes token transfer as agents only ingest targeted event data, and easily accommodates late-joining agents by replaying the event stream.C. Coordinator-Mediated (Hierarchical) ArchitectureThis is the default topology for frameworks like LangGraph. A central Supervisor or Coordinator agent explicitly manages the state flow, acting as a deterministic router.Mechanism: Workers receive a slice of state from the Coordinator, execute their task, and return the modified state back to the Coordinator.Bottleneck Risk: Because the Coordinator must synthesize the outputs of all subordinate agents to make subsequent routing decisions, it suffers from massive context accumulation. Scaling to 100+ agents requires creating nested sub-hierarchies to prevent the top-level supervisor's context window from failing.3.2 Context Distribution PatternsOnce the architecture is established, the physical payload of the context must be minimized during distribution.Full Broadcast (Inefficient): Any state mutation results in the entire context payload being transmitted to all agents. With a token cost scaling at $O(N \times \text{context\_size})$, this is economically unsustainable and acceptable only in micro-swarms (2-3 agents) executing very brief tasks.Diff-Based Push (Efficient): The optimal approach. When the global context changes, the system calculates a structural delta (e.g., using RFC 6902 JSON Patch format) and pushes only the exact modifications to the subscribed agents. The token cost scales at $O(N \times \text{diff\_size})$, slashing latency and inference costs.Pull-Based (On-Demand): Agents request context dynamically via tools. While highly token-efficient ($O(\text{requests} \times \text{context\_size})$), it introduces severe inference latency, as the agent must waste a conversational turn explicitly querying the database.Versioned Context: Each context object is cryptographically hashed to create a version ID. When communicating, the agent includes its local version ID. If a mismatch is detected by the server, the server calculates the delta between the agent's version and the current global version, transmitting only the necessary diff.Table 2: Multi-Agent Context TopologiesArchitectureIdeal Use CaseToken EfficiencyConsistency RiskScaling BottleneckBlackboardComplex, multi-stage reasoningHigh (Agents pull selectively)High (Concurrent writes)Database I/O / State lockingPub-SubLoosely coupled, reactive tasksVery High (Event-driven)Low (Eventual consistency)Message ordering / ObservabilityCoordinatorStrict, deterministic workflowsLow (Coordinator bloat)None (Strictly sequential)Coordinator token limits4. Part 3: Context Consistency and VersioningIn a parallelized multi-agent system, maintaining the integrity of the shared context is a formidable distributed systems challenge. If Agent A (a coder) and Agent B (a security reviewer) concurrently read and modify the same file state, naïve implementations will suffer from destructive overwrites and temporal hallucinations.4.1 Strategies for Concurrency and ConsistencyA. Lock-Based ConsistencyThe system employs mutual exclusion (Mutex) locks. When an agent requires access to a context node, it requests a lock.Mechanism: Agent A locks the file, reads, modifies, and writes back before unlocking. If Agent B attempts access, it is blocked.Trade-off: While this prevents race conditions and ensures linearizable reads, it completely destroys the parallelization advantages of multi-agent systems, introducing $O(N \times L)$ contention bottlenecks.B. Snapshot-Based (MVCC) ConsistencyMultiversion Concurrency Control (MVCC) allows agents to read a snapshot of the context at a specific timestamp.Mechanism: Agents execute work based on their local snapshot. If two agents attempt to commit modifications to the same snapshot, the system forces the second agent to re-evaluate its actions against the newly updated state.Trade-off: Reduces read-blocking but introduces "wasted work" where agents spend tokens generating solutions based on stale reads that are subsequently rejected during the commit phase.C. CRDT (Conflict-Free Replicated Data Types)
CRDTs represent the most sophisticated approach for LLM context consistency. CRDTs are data structures that guarantee Strong Eventual Consistency (SEC).Mechanism: Every agent maintains a local replica of the context. Agents can read and write to their local replica concurrently without locks. The underlying mathematics of the CRDT (utilizing operational transformations or state merging) ensures that once network synchronization occurs, all replicas deterministically converge to the exact same state without merge conflicts.Implementation (CodeCRDT): Recent frameworks like CodeCRDT instantiate this using the Yjs library and WebSockets. Agents observe a shared Y.Text document for code and a Y.Map for claiming "TODO" tasks. The system uses Last-Write-Wins (LWW) register semantics combined with logical clocks to prevent two agents from claiming the same task.Caveat: While CRDTs guarantee 100% syntactic convergence (no text corruption), LLMs can still generate semantic conflicts (e.g., both agents independently declare the same variable function). These semantic conflicts occur in roughly 5-10% of parallel operations and still require an overarching evaluator agent to resolve.4.2 Message History Propagation PatternsDefining how the context is structured within the LLM prompt is equally critical to how it is distributed over the network.Full History (The OpenAI Standard): Injecting the entire alternating array of user and assistant messages for every request. This is universally recognized as an anti-pattern for long-running agents, guaranteeing rapid context explosion.System + Messages (Anthropic Recommendation): Segregating immutable context (instructions, personas) into the system prompt, which is heavily cached by the provider, and passing only a truncated or summarized slice of the recent conversational turns in the messages array.Hierarchical Memory (AXORA Standard): The optimal pattern for production. The prompt structure maintains a strictly bounded size (~5,000 to 8,000 tokens) regardless of how many hundreds of turns have elapsed. It consists of:System: Immutable architecture, control flow, and tool schemas.Working Context (Blocks): Explicitly managed key-value pairs (e.g., current_goal, critical_files) updated via JSON patches.Recent Events: A strict rolling window of the last $N$ actions.Semantic Summaries: Highly compressed knowledge injected dynamically from the vector database.5. Part 4: Competitive Analysis of Industry ImplementationsA comparative analysis of existing frameworks reveals a spectrum of maturity regarding context management, ranging from naive message-passing to sophisticated virtualized memory systems.5.1 MemGPT / LettaLetta (formerly MemGPT) approaches the LLM as a central processing unit and implements memory management analogous to a traditional operating system.Context Sharing: Letta utilizes a Coordinator-Mediated approach heavily reliant on a persistent database. Multi-agent sharing is achieved through shared "Memory Blocks".Compacting Strategy: Letta relies on strict size limits for in-context memory blocks. It utilizes a Queue Manager to monitor token pressure; when the context approaches saturation, it triggers a "flush," forcibly evicting the oldest messages, recursively summarizing them, and moving the raw data into an external PostgreSQL/pgvector database (Archival/Recall storage).Consistency Model: Lock-based, relying on standard relational database transactions to prevent conflicting edits to shared memory blocks.5.2 LangGraphLangGraph is a low-level orchestration framework modeling multi-agent workflows as stateful, cyclical graphs.Context Sharing: Centralized graph state. The framework defines a rigid, explicit global state using Python's TypedDict or Pydantic schemas. This state is passed sequentially from node to node (agent to agent).Compacting Strategy: LangGraph does not natively auto-compact. Developers must explicitly define a "Trim Memory" node or write custom Reducer functions that dictate how new messages replace or summarize older lists to prevent bloat.Consistency Model: Reducer logic. LangGraph relies on the functional programming concept of reducers to safely aggregate state updates, inherently preventing race conditions by enforcing sequential super-step execution.5.3 AutoGenMicrosoft's AutoGen is optimized for conversational multi-agent interactions, excelling in multi-turn debates and collaborative refinement.Context Sharing: AutoGen employs a GroupChatManager that orchestrates a shared message thread (Pub-Sub via a Coordinator). It also supports ContextVariables—a shared dictionary allowing agents to store global parameters without repeating them in the chat history.Compacting Strategy: AutoGen primarily relies on simplistic list truncation (FIFO) or manual prompt engineering. It lacks native, advanced latent or memory-block compression, making it highly susceptible to context bloat in deep workflows.Consistency Model: Strict sequential message passing. Only one agent speaks at a time, dictated by the Group Chat Manager's routing algorithms, entirely eliminating concurrent state conflict at the cost of parallel speedups.5.4 CrewAICrewAI focuses on role-based, hierarchical team dynamics, combining short-term execution memory with long-term knowledge retention.Context Sharing: Unified memory architecture utilizing ChromaDB (for vector embeddings) and SQLite (for relational persistence). Agents inherently share a crew-level memory instance but execute tasks within isolated, private context scopes to prevent hallucination cross-contamination.Compacting Strategy: Entity extraction and RAG. CrewAI automatically extracts discrete facts and entities from task outputs, stores them as vectors, and utilizes semantic search to inject only highly relevant context into the next agent's prompt, effectively bypassing the need to pass full conversational transcripts.Consistency Model: "Cognitive" Consolidation. CrewAI employs an active consolidation logic layer that detects similar memories, identifies contradictions, and executes a planned merge/delete operation to ensure the retrieval database remains coherent.5.5 OpenHands (OpenDevin)Designed specifically for massive software engineering tasks, OpenHands represents the bleeding edge of event-sourced context management.Context Sharing: A strict event-sourced state architecture. A single conversation state object acts as the immutable source of truth, while individual agents act as stateless functional evaluators.Compacting Strategy: The LLMSummarizingCondenser. This rolling-window threshold system triggers automatically when history exceeds token limits. It aggressively condenses the middle of the event log into a CondensationEvent using LLM summarization, keeping the initial instructions and recent workspace actions pristine. This changes the cost curve of long operations from quadratic to linear.Consistency Model: Deterministic Event Replay. Because state is a function of an immutable sequence of events ($S_t = f(S_0, E_1, E_2,..., E_t)$), the system guarantees absolute consistency and the ability to "time travel" or rollback without conflict.5.6 DifyDify is an application orchestration platform that treats context primarily as an ETL (Extract, Transform, Load) problem for RAG pipelines.Context Sharing: Pipeline workflow state. Context is shared across nodes via strictly defined pipeline variables.Compacting Strategy: Dify relies on basic token pruning for conversations but excels in document compaction via its Summary Index. Instead of isolated chunks, Dify attaches a semantic summary to clusters of text chunks. This allows the RAG retrieval system to pull holistic, structured context rather than fragmented, disconnected strings, dramatically improving the LLM's high-level understanding of injected data.Consistency Model: Synchronous pipeline execution with document-level overwrite capabilities.Table 3: Competitive Analysis of Context ArchitecturesFrameworkTopology ParadigmNative Compaction EngineConsistency / ConcurrencyTarget Use CaseLettaOS Memory PagingMemory Block Self-EditingDatabase TransactionsLong-term memory assistantsLangGraphGraph / State MachineManual (Reducers/Trimming)Functional ReducersDeterministic enterprise workflowsAutoGenGroup ConversationMessage TruncationSequential Round-RobinMulti-agent debate & consensusCrewAIRole-based DelegationSemantic Extraction (RAG)Cognitive ConsolidationTeam-based task pipelinesOpenHandsEvent-Sourced LogRolling CondenserDeterministic Event ReplayDeep software engineeringDifyVisual RAG PipelineSummary IndexPipeline Variable StateKnowledge base QA apps6. Part 5: AXORA Architecture RecommendationsFor the AXORA platform to achieve production-grade reliability across 100+ turns with dozens of interacting agents, it must synthesize the most effective components analyzed in this research. The recommended architecture replaces rigid, centralized locking with distributed, eventually consistent structures.6.1 The Coordinator + CRDT Blackboard TopologyAXORA should implement a hybrid Coordinator-Mediated Blackboard utilizing Conflict-Free Replicated Data Types (CRDTs).Global Semantic State (Coordinator): The DADD (Domain Agent Distributed Director) Coordinator does not process raw interaction logs. It maintains a lightweight graph of active tasks, agent health metrics, and high-level milestones.Shared State (CRDT Blackboard): The actual operational data—shared documents, codebase states, and JSON parameter schemas—is hosted on an in-memory blackboard. By implementing this layer with a CRDT library (such as Yjs), AXORA enables true parallelization. Multiple worker agents can read and write to the blackboard simultaneously. The CRDT mathematics (e.g., Y.Map for key-value state and Y.Text for text generation) guarantee that all agent replicas will reach Strong Eventual Consistency (SEC) without requiring blocking Mutex locks.Diff-Based Event Bus: To eliminate the $O(N \times C)$ full broadcast token penalty, AXORA will utilize an event bus (e.g., Redis Pub/Sub). When the CRDT resolves a state mutation on the blackboard, the system generates an RFC 6902 JSON Patch. The Coordinator publishes this exact diff to the relevant topic, and only the agents subscribed to that topic update their local in-window context.6.2 Context Compaction IntegrationDrawing from OpenHands and Letta, AXORA's worker agents must utilize a strictly governed, hierarchical prompt structure to prevent context rot.Working Memory Blocks: The agent's prompt will contain a pre-allocated JSON structure acting as working memory. Agents are provided tools to emit structural patches to modify this block directly, maintaining their own running state.The Background Condenser: AXORA will implement a specialized background compaction pipeline mimicking the LLMSummarizingCondenser. A lightweight, highly distilled model (specifically tuned to avoid the Size-Fidelity Paradox ) continuously monitors the event log. When a worker's context breaches the 8,000-token limit, the condenser extracts the oldest 5,000 tokens of intermediate reasoning, compresses them into a highly dense semantic summary, and seamlessly replaces the raw logs in the agent's active context window.6.3 Integration with Tripartite Memory (R-14)This architecture integrates fluidly with the Tripartite Memory model defined in R-14:Episodic Memory (Chronological): The raw event stream excised by the Condenser is not deleted; it is flushed to an append-only SQLite database. If an agent requires deep historical specifics, it utilizes a retrieval tool to query this episodic vault.Semantic Memory (Factual): Extracted facts and finalized CRDT blackboard states are vectorized into ChromaDB. They are retrieved and injected into the agent's context only during relevant tasks, keeping the baseline prompt footprint minimal.Procedural Memory (Skills): Standardized tools and operational pipelines are stored structurally. If a procedural skill definition is updated by the Coordinator globally, the change is pushed as a versioned diff directly into the active worker's instruction block.7.
