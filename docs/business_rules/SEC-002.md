# SQL Injection Prevention

## Rule Definition

All database queries must use parameterized queries or prepared statements to prevent SQL injection attacks.

## Validation Criteria

1. **No String Concatenation** — Never concatenate user input into SQL strings
2. **Parameterized Queries** — Always use query parameters for user input
3. **ORM Usage** — Prefer ORM query builder over raw SQL
4. **Stored Procedures** — Use stored procedures for complex queries
5. **Least Privilege** — Database user has minimum required permissions

## Implementation Notes

- Use `sqlx` crate with compile-time query verification
- Enable `offline` mode for CI/CD without database
- Review all raw SQL in code reviews
- Use query logging to detect potential injection attempts

## Safe Query Example

```rust
// ✅ SAFE: Parameterized query
let user = sqlx::query_as::<_, User>(
    "SELECT * FROM users WHERE email = $1 AND active = $2"
)
.bind(&email)
.bind(true)
.fetch_one(&pool)
.await?;

// ❌ UNSAFE: String concatenation (DO NOT USE)
let query = format!(
    "SELECT * FROM users WHERE email = '{}'",
    email // VULNERABLE TO SQL INJECTION
);
```
