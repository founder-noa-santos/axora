# API Versioning

## Rule Definition

All API endpoints must be versioned to ensure backward compatibility and allow for controlled deprecation.

## Versioning Strategy

1. **URL Versioning** — Version in URL path (`/api/v1/...`)
2. **Semantic Versioning** — Follow semver (MAJOR.MINOR.PATCH)
3. **Deprecation Policy** — Minimum 6 months notice before removal
4. **Documentation** — Each version fully documented
5. **Testing** — All versions tested independently

## Version Format

```
/api/v{major}/resource
Example: /api/v1/users, /api/v2/users
```

## Implementation Notes

- Use middleware to extract and validate version
- Route to version-specific handlers
- Include deprecation headers in responses
- Maintain version compatibility matrix
- Provide migration guides for major versions

## Response Headers

```http
X-API-Version: 1.0.0
X-API-Deprecated: false
X-API-Sunset: 2027-03-16  # When this version will be removed
```

## Error Handling

```rust
pub enum ApiVersionError {
    UnsupportedVersion { requested: String, supported: Vec<String> },
    DeprecatedVersion { sunset: DateTime<Utc>, migration_url: String },
    InvalidVersionFormat,
}
```

## Version Lifecycle

```
1. Active → Fully supported
2. Deprecated → Still works, migration recommended
3. Sunset → Read-only, no new features
4. Removed → Returns 410 Gone
```

- Provide clear migration paths
