# Payment Processing

## Rule Definition

All payment transactions must be processed securely with proper validation, idempotency, and audit logging to ensure financial integrity.

## Validation Criteria

1. **Idempotency** — Each payment request must have unique idempotency key
2. **Amount Validation** — Payment amount must be positive and within limits
3. **Currency Validation** — Currency must be ISO 4217 compliant
4. **Card Validation** — Card numbers must pass Luhn algorithm check
5. **Fraud Detection** — Transactions must pass fraud scoring

## Implementation Notes

- Use `stripe` or `paddle` SDK for payment processing
- Store idempotency keys with 24-hour TTL
- Log all payment events for audit trail
- Implement webhook signature verification
- Never store raw card numbers (PCI-DSS compliance)

## Payment Flow

```
1. Create Payment Intent → Validate amount, currency
2. Process Payment → Idempotent operation
3. Handle Webhook → Verify signature, update status
4. Send Receipt → Email confirmation
```

## Error Handling

```rust
pub enum PaymentError {
    InvalidAmount,
    InvalidCurrency,
    CardDeclined(String),
    FraudDetected,
    DuplicateTransaction,
    WebhookSignatureInvalid,
}
```
