# 07. Domain Model and Business Entities

## Purpose

Map the business entities that now define OPENAKTA’s runtime.

## Executive Summary

OPENAKTA’s domain model is now broader than task orchestration alone. The key entities are agents, tasks, compressed context packs, provider requests/responses, MCP tool calls, audit events, semantic memories, episodic traces, procedural skills, and LivingDocs reconciliation artifacts.

## Core Runtime Entities

| Entity | Meaning |
| --- | --- |
| Agent | A registered execution participant with role and runtime status |
| Task | A unit of work with type, status, assignee, and dependencies |
| TaskAssignment | The typed instruction sent from coordinator to worker |
| ResultSubmission | Structured completion payload with token usage and optional patch receipt |
| WorkflowTransitionEvent | State transition record for observability and retries |

## Context and Compression Entities

| Entity | Meaning |
| --- | --- |
| ContextPack | Canonical task context envelope |
| `toon_payload` | Token-optimized serialized payload sent to the model boundary |
| MetaGlyphCommand | Symbolic prompt/control directive for compressed execution context |
| `latent_context` | Optional opaque latent side-channel for same-model experiments |
| RetrievalHit / ContextSpan / SymbolMap | Retrieved and targeted context units |

## Tooling and Security Entities

| Entity | Meaning |
| --- | --- |
| ToolDefinition | MCP-exposed tool contract |
| ToolCallRequest | Tool execution request over gRPC |
| ToolCallResult | Structured tool result with stdout/stderr/output |
| CapabilityPolicy | Action, scope, and timeout constraints |
| AuditEvent | Immutable record of allowed or denied tool use |

## Memory Entities

| Entity | Meaning |
| --- | --- |
| SemanticMemory | Embedded knowledge artifact such as synced docs or architectural facts |
| EpisodicMemory | Chronological trace of thoughts, actions, observations, successes, and failures |
| ProceduralSkill | Reusable skill artifact persisted under the skills root |
| PruningCheckpoint | Operational checkpoint for memory lifecycle work |
| UtilityScore | Procedural success/failure effectiveness signal |

## Governance Entities

| Entity | Meaning |
| --- | --- |
| MerkleTree | Repository change fingerprint |
| DocUpdate | Candidate documentation patch generated by reconciliation |
| ReconcileDecision | `Noop`, `UpdateRequired`, or `ReviewRequired` decision from LivingDocs |

## Business Meaning

The business entity model now treats memory, tool governance, and compressed context as first-class runtime primitives, not as future architecture notes.

## Open Ambiguities

- Legacy state abstractions still coexist with the newer memory stack.
- Procedural skill generation is real, but quality still depends on consolidation heuristics.
